## Key Observations

### EM vs VP gap вЂ” correct shapes, wrong specific names
`mapв†’unknown`: VP=99%, EM=10% вЂ” the model consistently generates a valid `Map<K,V>` shape but rarely guesses the exact same K and V. Same for `promiseв†’unknown` (VP=93%, EM=18%). Expected: exact-match is too strict a metric for complex generic types.

### `conditional_typeв†’unknown` вЂ” worst rule (56% acc)
Ambiguity: `unknown` can come from many rules (12вЂ“14f). The model must infer from context whether it is a conditional, indexed access, utility type, etc., all producing the same degraded literal.

### Underrepresented rules
- `react_component_props_with_refв†’any`: only **5 training pairs** вЂ” effectively untrained
- `readonly_arrayв†’unknown`: 4 val pairs вЂ” statistically unreliable
- `astro_*`: 2вЂ“3 pairs вЂ” not meaningful

### Overfitting after epoch 20
Best val 94.0% at epoch 20, final 89.2% at epoch 50 вЂ” a loss of ~5pp. No early stopping and no LR scheduler.

## Playground Run Results

After training, the model was tested on `SampleComponent.tsx` (deliberately degraded types). Results:

**Correct predictions вњ…**
- `React.SyntheticEvent` в†’ `React.MouseEvent` вЂ” perfect (rule #2)
- `useState<string>` в†’ `useState<"idle" | "copying" | "copied">` вЂ” inferred from the initial value `"idle"`, impressive
- `Map<unknown, unknown>` в†’ `Map<string, unknown>` вЂ” key correct, value not

**Hallucinations / false positives вќЊ**
- `label: string` в†’ `'horizontal' | 'vertical' | 'inline'` вЂ” **false positive**: `label` was a genuine `string` that should have been preserved; the model modified it incorrectly
- `variant: string` в†’ `'empty' | 'error' | 'warning'` вЂ” wrong values; expected `"primary" | "secondary" | "danger"`
- `Record<string, unknown>` в†’ `Record<string, LocaleModule>` вЂ” `LocaleModule` is a name memorised from training data, not present in this file
- `Set<unknown>` в†’ `Set<RouteWrite>` вЂ” same issue, `RouteWrite` is a hallucinated name

**Root cause of hallucinations:** the model memorises type names from training data instead of reasoning about the current file's context. Classic symptom of overfitting on a small, homogeneous dataset.

## Improvement Priorities

### рџ”ґ Critical вЂ” easy to fix

**1. Early stopping / save best checkpoint**
Trivial code change in `train.py`, guaranteed +5pp (89.2% в†’ ~94%).

**2. `--balance-rules`**
The flag already exists, just unused. Large rules (`html_specific_element`: 909 pairs) dominate every batch; small rules (`react_component_props_with_ref`: 5 pairs) are barely seen by the model.

### рџ”ґ Critical вЂ” requires data work

**3. False positives on `string_literal_unionв†’string`**
The biggest production-quality issue. The model cannot distinguish a degraded union (should be refined) from a genuine `string` (should be preserved). Fix: increase the negative ratio for this rule, or raise the global negative ratio from 25% to ~40%.

**4. Type name hallucinations (`LocaleModule`, `RouteWrite`)**
The model memorises specific names from training data. Two directions:
- More repos (more diversity)
- Higher dropout (regularisation)

### рџџЎ Important вЂ” requires pipeline changes

**5. Context radius = 0 в†’ too little context**
The extractor currently captures **0 surrounding lines**. The model sees only one line of code. Increasing to В±3 lines would give more signal вЂ” especially important for `utility_typeв†’unknown` and `string_literal_unionв†’string`.

**6. LR scheduler**
Loss oscillates 0.03вЂ“0.07 from epoch 15 with no LR decay. Cosine decay or ReduceLROnPlateau would stabilise the tail of training.

**7. `conditional_typeв†’unknown` disambiguation**
Add a signal to the prompt indicating whether the original span contained `extends`/`?`/`:`, to distinguish conditional types from other `в†’unknown` rules.

### рџџў Lower priority

**8. More data for underrepresented rules**
Add type-defs to the training dataset вЂ” they contain more `Partial`, `Pick`, `Omit` examples, which would improve `utility_typeв†’unknown` and `react_component_props_with_refв†’any`.

**9. Disable `react_component_props_with_refв†’any` in inference**
Only 5 training pairs вЂ” effectively untrained. Either add data or mute this rule in the locator until it has sufficient coverage.
