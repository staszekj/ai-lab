## Key Observations

### EM vs VP gap — correct shapes, wrong specific names
`map→unknown`: VP=99%, EM=10% — the model consistently generates a valid `Map<K,V>` shape but rarely guesses the exact same K and V. Same for `promise→unknown` (VP=93%, EM=18%). Expected: exact-match is too strict a metric for complex generic types.

### `conditional_type→unknown` — worst rule (56% acc)
Ambiguity: `unknown` can come from many rules (12–14f). The model must infer from context whether it is a conditional, indexed access, utility type, etc., all producing the same degraded literal.

### Underrepresented rules
- `react_component_props_with_ref→any`: only **5 training pairs** — effectively untrained
- `readonly_array→unknown`: 4 val pairs — statistically unreliable
- `astro_*`: 2–3 pairs — not meaningful

### Overfitting after epoch 20
Best val 94.0% at epoch 20, final 89.2% at epoch 50 — a loss of ~5pp. No early stopping and no LR scheduler.

## Playground Run Results

After training, the model was tested on `SampleComponent.tsx` (deliberately degraded types). Results:

**Correct predictions ✅**
- `React.SyntheticEvent` → `React.MouseEvent` — perfect (rule #2)
- `useState<string>` → `useState<"idle" | "copying" | "copied">` — inferred from the initial value `"idle"`, impressive
- `Map<unknown, unknown>` → `Map<string, unknown>` — key correct, value not

**Hallucinations / false positives ❌**
- `label: string` → `'horizontal' | 'vertical' | 'inline'` — **false positive**: `label` was a genuine `string` that should have been preserved; the model modified it incorrectly
- `variant: string` → `'empty' | 'error' | 'warning'` — wrong values; expected `"primary" | "secondary" | "danger"`
- `Record<string, unknown>` → `Record<string, LocaleModule>` — `LocaleModule` is a name memorised from training data, not present in this file
- `Set<unknown>` → `Set<RouteWrite>` — same issue, `RouteWrite` is a hallucinated name

**Root cause of hallucinations:** the model memorises type names from training data instead of reasoning about the current file's context. Classic symptom of overfitting on a small, homogeneous dataset.

## Improvement Priorities

### 🔴 Critical — easy to fix

**1. Early stopping / save best checkpoint**
Trivial code change in `train.py`, guaranteed +5pp (89.2% → ~94%).

**2. `--balance-rules`**
The flag already exists, just unused. Large rules (`html_specific_element`: 909 pairs) dominate every batch; small rules (`react_component_props_with_ref`: 5 pairs) are barely seen by the model.

### 🔴 Critical — requires data work

**3. False positives on `string_literal_union→string`**
The biggest production-quality issue. The model cannot distinguish a degraded union (should be refined) from a genuine `string` (should be preserved). Fix: increase the negative ratio for this rule, or raise the global negative ratio from 25% to ~40%.

**4. Type name hallucinations (`LocaleModule`, `RouteWrite`)**
The model memorises specific names from training data. Two directions:
- More repos (more diversity)
- Higher dropout (regularisation)

### 🟡 Important — requires pipeline changes

**5. Context radius = 0 → too little context**
The extractor currently captures **0 surrounding lines**. The model sees only one line of code. Increasing to ±3 lines would give more signal — especially important for `utility_type→unknown` and `string_literal_union→string`.

**6. LR scheduler**
Loss oscillates 0.03–0.07 from epoch 15 with no LR decay. Cosine decay or ReduceLROnPlateau would stabilise the tail of training.

**7. `conditional_type→unknown` disambiguation**
Add a signal to the prompt indicating whether the original span contained `extends`/`?`/`:`, to distinguish conditional types from other `→unknown` rules.

### 🟢 Lower priority

**8. More data for underrepresented rules**
Add type-defs to the training dataset — they contain more `Partial`, `Pick`, `Omit` examples, which would improve `utility_type→unknown` and `react_component_props_with_ref→any`.

**9. Disable `react_component_props_with_ref→any` in inference**
Only 5 training pairs — effectively untrained. Either add data or mute this rule in the locator until it has sufficient coverage.
