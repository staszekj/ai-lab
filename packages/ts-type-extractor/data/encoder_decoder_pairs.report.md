## Rule Coverage Quality

Raw `positive_pairs / total_annotations` would mix two unrelated causes of
"skip": (a) the type is already trivial (`string`, `unknown`, …) and can
never be degraded, and (b) the type is non-trivial but no rule targets it —
a real coverage gap. Addressable coverage rate isolates the second cause,
which is the only one actionable by writing more rules.

```
trivial_annotations       = count(typeText ∈ SIMPLE_TYPES)
                             SIMPLE_TYPES = { void, boolean, string, number,
                                              unknown, any, never,
                                              undefined, null }

addressable_annotations   = total_annotations − trivial_annotations

addressable_coverage_rate = positive_pairs / addressable_annotations
                     12.1% = 27238 / 225190
```

- Trivial annotations: **95575** (already string/number/boolean/any/…, never degradable — always exact `SIMPLE_TYPES` matches, computed straight from the skipped-types list)
- Addressable annotations: **225190** (total − trivial)
- **Addressable coverage rate: 12.1%**

Reads as: *of the annotations that were actually worth degrading, how many
did the current 17 active rules capture?* The gap between 12.1% and 100% is
the real headroom — see "Top skipped types (non-trivial only)" below for
where to add rules next.

## Notes

- Dataset was generated from usage repositories.
- Output file is ready for `refiner-train`.
- `MUTED_RULES` in `degrade.ts` is the only lever that removes examples from
  training data — it must be applied *before* regenerating
  `encoder_decoder_pairs.jsonl`. The mirror set in `ts_type_refiner/rules/validators.py`
  only filters the `VALIDATORS` dict used by `evaluate_exact_match` (training-time
  metric reporting) and `infer.py` (inference acceptance gate) — it has no
  effect on which rows `TypeRefinerDataset` loads or trains on. `refiner-locate.ts`
  carries a third, independent copy (`MUTED_RULE_KEYS`, by short `key` not full
  `rule` name) that gates inference-time candidate discovery — all three must be
  kept in sync by hand.
- Muting a rule can **redirect** a small number of examples to a different,
  lower-priority rule instead of just deleting them: `degradeType()` tries
  rules in array order and returns the first non-muted match, so an
  ambiguous type text that used to match a muted rule first now falls
  through to whichever active rule matches next. Muting `conditional_type→unknown`
  (round 2) redirected 7 examples into `utility_type→unknown` and 1 into
  `promise→unknown` (both match `T extends X ? A : B` shapes wrapped in a
  utility type / `Promise<…>`) — negligible here (8 / 27512), but worth
  checking when muting a rule with broader regex overlap.

Quick terminology:
- positive pair: input contains a degraded type, target contains the precise type
- negative pair: input and target intentionally preserve precision (signal: "do not change anything")
- recoverability: fraction of non-trivial identifiers in the target that also appear in the `siblings` context (how much of the answer is "nearby" vs. must be recalled/hallucinated)
